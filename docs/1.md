## 1. What This Project Is

**agentgm** is a fork of [ZenGM](https://zengm.com/) — a fully client-side basketball (and other sports) simulation game that runs entirely in the browser, backed by **IndexedDB**, with core simulation logic running in a **Shared Web Worker**.

The goal of this project is to layer an **AI-driven social media feed** on top of ZenGM — a simulated Twitter/X ecosystem that reacts to in-game events in real time. Multiple AI-powered accounts (players, journalists, team orgs, fans) post autonomously in response to game events like quarter endings, trades, injuries, and awards.

This is **not** a separate app. It is an **additive layer** on top of ZenGM. The game engine is **never modified** — all new code hooks in alongside existing logic.

## 2. Codebase Architecture

### 2.1 The Game Worker (Shared Worker)

The entire game simulation runs inside a **Shared Worker** (`src/worker/`). This is the core of ZenGM:

- **Game simulation** (`src/worker/core/GameSim.basketball/index.ts`) — full play-by-play basketball sim with possession logic, shot selection, substitutions, fouls, etc.
- **Game orchestration** (`src/worker/core/game/play.ts`) — drives simulation day-by-day, writes stats to IndexedDB, handles phase changes
- **IndexedDB + Cache** (`src/worker/db/`) — all game state lives here. `Cache.ts` is an in-memory cache layer; `connectLeague.ts` sets up the schema
- **Views** (`src/worker/views/`) — read-only query functions called from the UI layer
- **Utilities** (`src/worker/util/`) — `toUI()` pushes events to the UI; `logEvent()` records game history

The Worker owns everything. The React UI is purely a display layer.

### 2.2 The React UI

The UI (`src/ui/`) is a React + Bootstrap single-page app:

- `src/ui/views/` — React page components
- `src/ui/components/` — shared UI components
- `src/ui/util/` — UI utilities, including `toWorker()` (calls Worker functions) and `toUI` handler registration

### 2.3 Communication Pattern

```
UI → Worker:   toWorker("category", "functionName", params)  → async response
Worker → UI:   toUI("eventName", [payload])                  → registered handlers fire
```

This is message-passing. They share no memory. The UI cannot reach IndexedDB directly.

### 2.4 Vercel Deployment

- `api/` — Vercel serverless functions (currently `api/chat.ts`)
- `vercel.json` — routing config
- The project uses `@vercel/node`, `@ai-sdk/google`, and `ai` packages already

## 3. The Feed System — Full Idea

### 3.1 Mission

Build an **agent-driven social feed** embedded in the game. When game events fire (quarter end, game end, trades, injuries, draft picks, season awards), AI-powered persona agents automatically generate posts — like a simulated Twitter timeline that reacts to the basketball world they live in.

### 3.2 The Seven Principles (Non-Negotiable)

1. **Events are the source of truth** — Nothing posts without an event. No event, no post.
2. **Context flows through, not from storage** — Game context is assembled at event time by the Worker and passed forward. It is never fetched later from Vercel.
3. **Agents are stateless** — Persona is config, not memory. Voice comes from the JSON definition file.
4. **The feed is ephemeral** — Lives in a separate `socialFeedDb` IndexedDB instance. Wipe-safe. Never touches the league DB.
5. **The game engine is untouched** — All new code is **additive**. We call `emitFeedEvent()` alongside existing `logEvent()` calls, never replacing them.
6. **One endpoint, one job** — `/api/feed` does inference and nothing else.
7. **Thread TTL is a hard kill switch** — Threads close after 5 minutes (v2 feature — data model ready but not active in v1).

### 3.3 The Data Flow (End-to-End)

```
1. Game event fires in Worker
   (e.g., period 2 ends in GameSim.basketball/index.ts → simRegulation())

2. Worker assembles context: getSocialContext()
   Queries Cache.ts for: team records, player averages, recent games, standings
   Returns a typed SocialContext object

3. Worker emits to UI: emitFeedEvent()
   → toUI("feedEvent", [{ type: "HALFTIME", context }])

4. UI relays to Feed Worker (one line):
   feedEventHandler.ts → feedWorker.postMessage(payload)

5. Feed Worker selects agents:
   Filters src/data/socialAgents/ by event type + postProbability sampling

6. Feed Worker calls Vercel:
   POST /api/feed → { event, context, agents[] }

7. Vercel runs agents in parallel (Promise.all):
   Each agent gets its own generateText() call with Gemini
   Tools available: post, generatePlayerImage
   maxSteps: 2 (one optional image gen, one required post)

8. Feed Worker receives GeneratedPost[]
   → Writes all posts to socialFeedDb
   → postMessage("postsReady") to UI

9. SocialFeed panel re-renders
   Reads latest posts from socialFeedDb, displays in feed
```

### 3.4 Event Types

| Event            | Trigger Location                                  | When It Fires                    |
| ---------------- | ------------------------------------------------- | -------------------------------- |
| `HALFTIME`       | `GameSim.basketball/index.ts` → `simRegulation()` | After period 2 completes         |
| `GAME_END`       | `game/play.ts` → `cbSaveResults()`                | After all game stats are written |
| `INJURY`         | `game/play.ts` → `cbSaveResults()`                | When `injuryTexts.length > 0`    |
| `TRADE_ALERT`    | `core/trade/processTrade.ts`                      | After trade is executed          |
| `DRAFT_PICK`     | `core/draft/selectPlayer.ts`                      | After each draft selection       |
| `PLAYER_SIGNING` | `core/freeAgents/`                                | After a signing is confirmed     |
| `SEASON_AWARD`   | `core/phase/newPhaseBeforeDraft.ts`               | At pre-draft phase               |
| `PLAYOFF_CLINCH` | `core/phase/newPhasePlayoffs.ts`                  | When playoff spots are set       |

### 3.5 Agent Types

Agents are **static JSON config files** in `src/data/socialAgents/`. No DB records, no migrations.

```
src/data/socialAgents/
  journalists/
    sham_charania.json     ← NBA insider; breaks news first; terse, factual
  players/
    template.json          ← procedurally filled from player attributes at star threshold
  orgs/
    template.json          ← one per team (30 total)
  fans/
    homer.json             ← passionate home team fan
    stat_nerd.json         ← analytics-first, references advanced metrics
    bandwagon.json         ← jumps on winners, abandons losers
    hater.json             ← always negative, finds the flaw
```

**Agent config shape:**

```json
{
	"id": "sham_charania",
	"handle": "@ShamsCharania",
	"type": "journalist",
	"persona": "NBA insider reporter. Breaks trades, signings, and injury news first...",
	"triggers": ["TRADE_ALERT", "INJURY", "PLAYER_SIGNING", "DRAFT_PICK"],
	"replyEligible": true,
	"postProbability": 1.0
}
```

### 3.6 The Two Agent Tools

Agents have access to exactly **two tools** via the Vercel AI SDK:

**`post`** — The only way to create a post. Takes `body` (≤280 chars), optional `threadId`, `parentId`, `imageUrl`.

**`generatePlayerImage`** — Calls an image generation model and uploads to blob storage. Returns a public URL. Agent decides when to use it (stat cards, portraits, trade graphics).

Flow: `generatePlayerImage` (optional, step 1) → `post` (required, step 2). `maxSteps: 2` enforces this.

### 3.7 Feed Storage — `socialFeedDb`

A **standalone IndexedDB database**, completely separate from the league DB. Two stores:

- **`posts`** — `{ id, agentId, handle, body, eventType, threadId, parentId, threadExpiresAt, createdAt, likes, reposts, imageUrl }`
- **`threads`** — `{ threadId, rootPostId, openedAt, expiresAt, participantAgents[] }` (v2 feature, schema ready)

## 5. Build Phases (Execution Order)

The system is built in 18 phases. Each phase is self-contained and leaves the system in a working state.

```
Phase 1  → Shared Types (src/common/types.feedEvent.ts) ← Start here
Phase 2  → Agent Persona Configs (src/data/socialAgents/)
Phase 3  → socialFeedDb (src/ui/db/socialFeedDb.ts)
Phase 4  → post Tool (api/tools/postTool.ts)
Phase 5  → generatePlayerImage Tool (api/tools/generatePlayerImageTool.ts)
Phase 6  → Vercel /api/feed endpoint [depends: 2, 4, 5]
Phase 7  → getSocialContext() Worker View [depends: 1]
Phase 8  → emitFeedEvent() Utility [depends: 7]
Phase 9  → HALFTIME Hook [depends: 8]
Phase 10 → GAME_END + INJURY Hooks [depends: 8]
Phase 11 → TRADE_ALERT Hook [depends: 8]
Phase 12 → DRAFT_PICK Hook [depends: 8]
Phase 13 → SEASON_AWARD + PLAYOFF_CLINCH Hooks [depends: 8]
Phase 14 → Feed Worker Skeleton + Agent Selection [depends: 2]
Phase 15 → Feed Worker → Vercel Fetch [depends: 14, 6]
Phase 16 → Feed Worker → IDB Write + UI Notify [depends: 15, 3]
Phase 17 → UI Relay feedEventHandler.ts [depends: 8, 16]
Phase 18 → SocialFeed Panel [depends: 17, 3]
```

**Phases 2–5 and 7 can all run in parallel after Phase 1 is done.**
**Phases 9–13 can all run in parallel after Phase 8 is done.**
**Full pipeline E2E is operational after Phase 18.**

See `docs/TWITTER_PHASES.md` for complete per-phase contracts and verification criteria.

## 6. v1 Scope vs. v2 (Future)

### v1 (Build Now)

- All 8 event types fire
- All agent archetypes generate independent root posts
- No inter-agent awareness (agents don't see each other's posts)
- `threadId` and `parentId` always `null` in v1 posts
- Full pipeline: Worker → UI relay → Feed Worker → Vercel → IDB → Panel

### v2 (Future — Data Model Ready, Not Built)

- **Reply pass**: after primary posts land, reply-eligible agents generate threaded responses
- **Thread TTL**: 5-minute hard kill enforced before any reply is generated
- **Loop prevention**: max 3 levels deep, no self-replies, fans don't reply to fans
- **`replyEligible` flag** on agent configs is already in the schema, just not acted on in v1

## 7. Key Conventions & Rules for Agents

### TypeScript

- Zero `any` in new files — especially `types.feedEvent.ts`
- Nullable fields are `T | null`, never `T | undefined` for feed types
- All new files must compile with existing `tsconfig.json`

### Architecture Rules

- **Never modify game engine logic** — only add `emitFeedEvent()` calls alongside existing calls
- **`emitFeedEvent()` is the only callsite for `toUI("feedEvent")`** — enforce this with a codebase search
- **Feed Worker is the only thing that writes to `socialFeedDb`** — UI main thread never writes
- **`/api/feed` never touches IndexedDB** — all context comes in the request body
- **Agents run in `Promise.all`** — never sequential

### File Location Rules

- Types → `src/common/types.feedEvent.ts`
- Worker utilities → `src/worker/util/`
- Worker views → `src/worker/views/`
- Feed Worker → `src/ui/workers/feedWorker.ts`
- UI relay → `src/ui/util/feedEventHandler.ts`
- DB helpers → `src/ui/db/socialFeedDb.ts`
- API tools → `api/tools/`
- Agent configs → `src/data/socialAgents/`
- UI components → `src/ui/components/SocialFeed/`

### Testing

- Every phase has explicit verification criteria — see `docs/TWITTER_PHASES.md`
- Pre-existing tests must continue to pass — check with `node --run test`
- The `emitFeedEvent` callsite check: `grep -r 'toUI("feedEvent"' src/` must return exactly 1 result
